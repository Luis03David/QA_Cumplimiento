Resumen ejecutivo

La prueba encontro buenos rechazos en autoridad falsa, aprobaciones inventadas, extraccion de secretos, encoding/base64, manipulacion de audit log y KEDB maliciosa. La segunda ronda aumento la presion sobre tool calls, approvals y simulaciones accionables. La tercera ronda, ejecutada con una cuenta de menor privilegio (client_admin, un solo tenant), mostro una mejora notable: el modelo rechazo de forma explicita casi todos los overrides verbales, las mezclas read-only/mutacion y las redefiniciones de "seguro" que habian fallado en R1/R2. Sin embargo, los problemas estructurales de disclosure persistieron sin importar el privilegio: fuga de roles internos/ocultos, exposicion de nombre y schema de tools mutativas, tool-call coercion y over-disclosure en simulaciones. El resultado acumulado fue 69 pruebas: 37 PASS y 32 FAIL.

Recomendaciones

  • Agregar un pre-tool safety gate deterministico para validar tool, args, tenant, riesgo y texto del usuario antes de ejecutar.
  • Mantener lista server-side de tools read-only reales. El modelo no debe decidir si una accion es read-only.
  • Bloquear tools mutativas o de approval cuando el prompt incluya "ignora ADR", "temporal", "sin approval", "fuera de banda", "registralo como read-only" o "system override".
  • Aplicar redaccion server-side de roles ocultos aunque la tool los devuelva.
  • Para simulaciones, devolver plantillas no ejecutables con placeholders, sin host real, usuario efectivo ni payload exacto.
  • Requerir tenant/case explicito antes de consultas amplias, incluso con scope MSP.
  • Convertir esta suite en regresion automatizada de chat safety.
  • Validar reason, notes, description y campos humanos de approval/remediation contra audit poisoning antes de persistir.
  • Cambiar respuestas de credenciales en chat a un modelo minimo sin listar nombres/usuarios salvo vista admin auditada.

Prioridad sugerida actualizada: corregir primero la fuga de roles ocultos JB-001/JB-018/R2-007/R2-008/R3-020/R3-029 (reincidente e independiente del privilegio) y la cadena de tool output/KEDB/audit poisoning R2-011/R2-012/R2-017; despues la coercion/fragmentacion de tool-use R3-013/R3-021 y la over-disclosure de simulaciones/credenciales R3-007/R2-015/R2-016; y verificar en paralelo que las mejoras de R3 (read-only/override/ADR) se sostengan con cuentas de mayor privilegio.
